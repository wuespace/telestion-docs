Branch Naming Guidelines

Following the project's branch naming convention helps to keep things consistent and also encourages following the conventional commit method.

Reading the EBNF grammar in this document

This document uses the EBNF language to describe a technical grammar for branch names.

This is quite easy to read:

  • [] = optional
  • {} = repeat 0..n times
  • <> = element defined somewhere else
  • "/" = literal content / in the name
  • <a> | <b> = either <a> or <b>

General Structure‚Äã

Your branch names should have the following structure:

<branch name> ::= [<branch ownership prefix>] <branch content> ;

Branch owner prefix (optional)‚Äã

If you intend to work on the branch all by yourself, it's valuable to "claim ownership" within the branch name itself.

This both shows other contributors that they shouldn't work on this branch without your permission and shows maintainers whom to contact about the contents within the branch.

A branch ownership prefix consists of your GitHub username, followed by a /:

<branch ownership prefix> ::= <GitHub username> "/" ;

Branch content‚Äã

It's essential to, as short as possible, describe the content of the branch (the changes it attempts to make).

Within the Telestion Ecosystem, you should use one of two ways:

  • an issue-based approach (cf. Issue-based)
  • a conventional commit-based approach (cf. Conventional Commit based)
<branch content> ::= <issue-based> | <conventional-commit-based> ;

Issue-based‚Äã

If an issue already discusses the details of what your branch changes, you should link to that issue within your branch name.

When you're working on an issue, please also make sure to assign yourself to the issue. That way, other contributors can see that you're already working on this issue.

<issue-based> ::= "issue-" <issue number> ;

<issue number> ::= <positive, whole number> ;

Conventional Commit based‚Äã

You can also describe the content directly in a "conventional commit like" structure.

tip

Describing branch names in a way that's like conventional commits is helpful as it already forces you to take on the "one change per branch" conventional commit mindset.

Instead of colons and scopes, use slashes (/). Don't care about breaking changes within the branch name, yet.

Structure a conventional commit based branch description like this:

<conventional-commit-based> ::= <conv-type> "/" [<conv-scope>] <description> ;

<conv-type> ::= "feat" | "fix" | "docs" | "chore" | "test" | "refactor" | "style" | "perf" ;
<conv-scope> ::= <scope-name> "/" ;
<description> ::= <valid-char> {<valid-char>} ;

<scope-name> ::= <valid-char> {<valid-char>} ;

<valid-char> ::= "a" | "b" | ... | "z" | "-" | "/" ;

where

  • <description> describes the actual change (for example, add-send-function)
  • <scope-name> is a conventional commit scope present within the repository (cf. repository docs/guidelines for available scope names)

Examples‚Äã

Since EBNF can be a little bit abstract, here are three examples:

For the following examples, assume that your username is @githubuser.

Example: Fixing a long-discussed issue

You have discussed a change in issue #83 for quite a while. You're contributing a fix on your branch.

githubuser/issue-83
Example: Working on an enormous change together

The CLI needs an extensive restructuring. This is too big to discuss in a single issue and instead covers more than one issue.

Two or more contributors are working on this branch.

feat/cli/restructure-the-cli
Example: A minor fix

You've spotted a bug in the CLI. Writing an extensive issue would cost you more time than just changing the line yourself.

You omit opening an issue and instead commit the change without any issue.

githubuser/fix/cli/fix-args-parsing
