Unexpected Error Message Structure

NADAD: Unexpected Error Message componentsâ€‹

Unexpected error messages should follow the NADAD structure:

Notifyâ€‹

Developers appreciate directly getting information, even if it's information that they don't like. You should notify them what caused that unexpected error and where it occurred:

Unexpected error while trying to perform a network request.

Apologizeâ€‹

The fact that you didn't expect for this to happen means that it's you who messed up. Instead of trying to blame the developer who uses your library, admit your mistake and apologize:

We're sorry, it looks like we got an unexpected error.

Describeâ€‹

Give as good of an explanation as you can to give the other developer a general understanding of what happened. Maybe, this can help them solve the error themselves.

We had already validated the requested address, but when trying to request the
resource, the address could no longer be parsed.

Actionâ€‹

It's your mistake. Thus, it's up to you to fix it.

Provide instructions on what the developer should do so that you can fix the mistake. In other words: Provide instructions to, with minimal effort, file a bug:

Please help us fix the mistake by filing an issue at https://github.com/wuespace/telestion-core/issues/new
and include the entire error message (including the details below) and we'll have a look as soon as we can.
Thanks in advance!

Detailsâ€‹

Include any details you might need to find the mistake and either fix a bug or at least provide an actionable error message:

Details
=======
requestAddress: "http://noejow"
telestion-core version: v0.3.0
